⚙️ Graph Versioning v1 — Branch-Read Performance Investigation

pokec_small (10k vtx / 121k edges) · HEAD 9659d6e0b RelWithDebInfo · 10-run paired branch/main ratios · worst (~25%) & realistic (~1.3%) churn · 2026-07-14

Bottom line. Branch-read overhead has two structural cost centers, both main-result-bound (they do NOT shrink with a smaller branch): a scan path (wide index scans / aggregations, ~2.3–3.2× at realistic churn) and a traversal path (shortest-path ~4×, deep expansion ~3.3×). Of the levers investigated, exactly one is both cheap and confirmed — entry-value exposure (~½ the index-scan overhead). The traversal cost is architectural (per-hop gid re-resolution), not cheaply fixable. A fork-epoch base-retention design is feasible but not worth its commit-path cost.

Method (recap)

  • One instance; Session M on main, Session B on branch bench, separate single-connection drivers (checkout is connection-scoped).
  • Per run, M vs B measured interleaved per parameter; headline = per-run ratio branch/main, distribution across 10 runs (cancels machine transients → tight ranges).
  • Two churn profiles: worst ≈ 25% of vertices touched (COW 20% + 2% delete + split-adjacency hot set + native), realistic ≈ 1.3% (COW ~80, ~20 delete, 30 hot, 50 native).
  • Perf theories tested with destructive A/B probes (skip the suspected cost, measure the delta) since hardware PMU is unavailable on this VM.

Key finding: overhead is main-bound, not churn-bound. The heavy queries barely move from 25%→1.3% churn — they scan/traverse main's rows regardless of how little the branch changed. A realistic (small) branch does not make scan/traverse-heavy reads cheap. Only the split-adjacency cases shrink (fewer branch-added edges).

Levers investigated (all measured, none assumed)

SHIPPED 1 · Streaming merge (index scans)

The branch index-scan merge collected both streams (diff ∪ main@fork) into one vector and std::stable_sort'd it — but both streams are already index-ordered. Replaced with a streaming 2-way merge on the index value comparator (ASC/DESC-aware; B1 dropped-index fallback still sorts). Worst-case wide index scan 5.48× → 4.52× (~19% branch-side). In the working tree, unit-green (82/82, 10/10); not committed.

REVERTED — wash 2 · O(1) dedup set

Replaced the per-row FindVertex(g,NEW) skiplist COW-probe with an O(1) unordered_set<Gid>. No measurable change (index scan 2.068→2.060ms). The diff skiplist is small and cache-friendly; the dedup probe was never the bottleneck. Correct + asymptotically better, but perf-neutral at these scales → reverted to keep the diff minimal.

CONFIRMED → BUILD 3 · Entry-value exposure (index scans)

The merge re-reads each row's ordering key via GetProperty(View::OLD) — an MVCC read — even though the index entry already stores the value. A destructive probe (skip GetProperty, return a constant key) dropped the worst-case index scan 2.068ms → 1.121ms = ~46% of branch time, ratio 4.52× → 2.35×; controls stayed flat (clean isolation). Fix: expose the index entry's stored values through the scan iterator (both diff + historical sides) and use them directly. No MVCC change, no second copy — a storage-iterator change. This is the one cheap, confirmed win.

PROBE-DISPROVEN 4 · ResolveEdges union (traversals)

Hypothesis: the per-hop union machinery (diff FindVertex + seen dedup map + eager vector) drives the ~4× traversal cost. Probe stripped the entire union apparatus (kept only the historical read + O(1) tombstone filter): shortest-path stayed flat, 4.06× → 4.09×; clean expansions flat. So the union machinery is not the cost. The real traversal cost is architectural: per hop the branch does historical_base_->FindVertex(gid, OLD) — a skiplist re-resolution of the vertex by gid — then reads its edges through the historical accessor and materializes an eager vector<EdgeAccessor>, whereas main follows the raw neighbour Vertex* and streams. The overlay resolves by gid because a vertex may be COW'd or historical (it can't pointer-follow across the two stores). No cheap lever here — a real attack needs cross-store pointer-following (a much larger redesign). Traversal overhead accepted as architectural for v1.

FEASIBLE — but shelved 5 · Fork-epoch base-retention (user's design)

Retain each COW'd object's fork-value as a distinct MVCC version stamped at a reserved low "fork epoch" so HistoricalAccess(fork_epoch) on the diff engine yields a value-aligned suppression stream (iterator-2) for a 3-way merge — killing both the dedup probe and the property re-read on the scan path.

Feasibility corrected to TRUE. An earlier "infeasible — monotonic timestamp" read was incomplete: a version's visibility ts is stamped from the per-accessor commit_timestamp_ (storage.cpp:1283→:1448), separable from the shared timestamp_++. A CommitArgs::make_base_seed(F) variant can stamp a base at a low F without advancing the counter; GC=NONE + durability-off remove the horizon/LDT read-path concerns. (An alternative "open uncommitted base transaction" variant is dead — blocked by PrepareForWrite's write-write conflict against another still-open txn.)

Shelved on cost/benefit: invasive commit-finalization surgery (new CommitArgs variant + per-txn gating for the LDT assert + distinct low-F allocation + capture/replay re-seed + 3-way merge), it helps only the scan path (redundant with the far cheaper entry-value change), and does nothing for the dominant traversal cost. Its non-perf benefits (SHOW BRANCH DIFF "before", MERGE base, self-containment) are already served by the pinned main@fork.

Net stack & recommendation

Worst-case wide index scan: 5.48× (baseline) → 4.52× (streaming, shipped) → ~2.35× (with entry-value, confirmed-achievable).

  • Build entry-value exposure — the only cheap, confirmed lever; ~halves the wide-index-scan overhead.
  • Accept traversal overhead (~3–4×) as architectural for v1 — it's per-hop gid re-resolution, inherent to the overlay; document it, don't rush a cross-store redesign.
  • Do not build base-retention — feasible but redundant-with-entry-value on scans and useless for traversals.
  • Ship the streaming merge (already in tree, green).

Bug found during this work (unrelated to perf)

C3 — re-checkout/restart crash (HIGH, pre-existing, file separately). CaptureBranchCommit (branch_reconstruction.cpp:243) iterates a transaction's deltas in forward_list slab order (capacity 292, emplace_front on overflow) → reversed across slab boundaries. A single branch transaction with >292 deltas can capture a SET before its CREATE; on re-checkout, replay hits the mutation before the create → MG_ASSERT (branch_engine.cpp:221) → SIGABRT. Reconciliation test: a 1-vertex COW re-checkout is fine; a 150-vertex single-transaction COW crashes. Fix: iterate the transaction's deltas chronologically across slabs. This is what made the first (25%-churn, ~2000-vertex-per-txn) benchmark attempt crash — not a COW-semantics flaw.

Caveats

Single-instance, single-branch, IN_MEMORY_TRANSACTIONAL, pokec_small, read-only overhead. Absolute ms are hardware-specific; the ratios are the portable result. Not covered: write throughput, MERGE cost, concurrency, replication, larger datasets (scan/traversal ratios are expected to hold or grow with graph size since they're main-bound).
