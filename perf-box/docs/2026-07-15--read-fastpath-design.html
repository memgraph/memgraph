Branch read-path fast-path — design

Graph Versioning v1 · src/query/vertex_accessor.cpp · target: full-graph aggregation 2.26× + all branch property/label reads · 2026-07-15

Thesis. Every branch-mode property/label read currently pays a per-row ResolveVertex(gid, view) — a diff-engine skiplist probe + historical resolve — even for un-branched main objects. Extend the already-shipped branched-bit fast path (InEdges/OutEdges/To/From) to the five read methods so an un-branched main object reads impl_ directly. No layout change, mutation-safe, byte-identical result.

The overhead (measured)

A destructive probe (read plain historical only, no branch machinery) took full-graph aggregation from 2.26× → 1.01×. So the ~1.26× is the branch wrapper cost, split between (a) the UnionVerticesIterable two-stream gid-merge in db_accessor.hpp and (b) the per-row ResolveVertex in the read methods below. This change attacks (b); the follow-up measurement tells us how much of the 1.26× is (b) vs (a).

Current per-read path (all 5 methods identical shape)

storage::Result<PropertyValue> VertexAccessor::GetProperty(View view, PropertyId key) const {
  if (branch_ctx_ != nullptr) {
    if (auto resolved = branch_ctx_->ResolveVertex(impl_.Gid(), view)) {   // <-- skiplist probe PER ROW
      return resolved->GetProperty(key, view);
    }
    if (view == View::NEW && branch_ctx_->IsVertexTombstoned(impl_.Gid()))   // S2 guard
      return std::unexpected{Error::DELETED_OBJECT};
  }
  return impl_.GetProperty(key, view);
}

Why it is byte-identical to the slow path

  • For an un-branched, historical-resident object, ResolveVertex misses the diff engine and resolves the same gid on the historical side → the same Vertex* that impl_ already holds. So impl_.GetProperty(key, view) ≡ resolved->GetProperty(key, view). Read at the caller's view (not forced OLD) to be exactly the slow path's return.

Safety

hazardwhy fast path is safe
Mutation / COW-gating (the blocker that killed the plain-accessor idea)branch_ctx_ stays non-null. Mutators never read the bit — they unconditionally CowIfNeeded(). Only READ methods consult it. No mutator behavior changes.
HIGH-2 stale copy (MATCH (n) SET n.x=2 RETURN n.x)The SET's COW flips branched() → true and populates the diff engine. The subsequent RETURN read (even on a stale value-copy) fails the bit test → takes the re-resolve slow path → picks up the diff copy. Correct.
S2 tombstone (read-after-delete-in-command)TombstoneVertex calls MarkMainObjectBranched → branched()==true → fast path skipped → the View::NEW tombstone guard runs. Correct.
Stale-false bitImpossible during a live branch: the bit is set under the object lock at COW/tombstone time, and fast-discard is disabled while a fork-pin is held (so the one unlocked delta writer can't race the set). Restart re-seeds. Only stale-false could be wrong; it can't occur.

Scope

Five methods in vertex_accessor.cpp: Labels, HasLabel, Properties, GetProperty, GetPropertySize. One TU + relink (cheap rebuild). Edge-side (edge_accessor.cpp) reads: separate follow-up if measurement shows edge-property reads matter (shortest-path w/o edge props doesn't hit them).
