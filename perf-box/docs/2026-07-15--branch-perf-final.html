What each change did (measured, not assumed)

1 Streaming merge — index scans

Branch index-scan merge was collect-both-then-std::stable_sort; both streams are already index-ordered, so it's now a streaming 2-way merge. Wide index scan 5.48×→4.52× (worst).

2 Entry-value exposure — index scans

The merge re-read each row's ordering key via GetProperty(View::OLD) though the index entry already stores it. Exposing the entry's values through the iterator (zero-copy view) removed that per-row MVCC read — probe-confirmed ~46% of branch scan time. Index scan → 1.76× realistic.

3 Branched-bit filter — the root fix

A single monotonic "touched by any branch" bit in a spare tag bit of each main Vertex/Edge's packed delta pointer (zero size cost; same pattern as the existing deleted() bit). Set under the object lock whenever a branch COWs/tombstones a fork object; re-seeded from the changelog at checkout; never cleared in v1. It's a conservative filter: false ⇒ definitely-not-shadowed ⇒ take a fast path; true ⇒ existing per-branch reconciliation. This inverts the overlay's cost asymmetry — work becomes proportional to the branch, not to main — and is correctness-safe by construction (only a stale-false could be wrong, and the restart re-seed prevents it; stale-true just costs a slow path).

  • Scan fast-path: an un-branched main row skips the dedup/tombstone probe → index scan 1.76×→1.35×.
  • Traversal fast-path (OutEdges/InEdges): skips ResolveEdges' historical∪diff union for un-branched sources. Alone this did almost nothing (shortest-path 3.86×) — because the edge-list retrieval was never the traversal bottleneck.

4 To/From endpoint fast-path — the traversal fix

The real per-hop cost: every traversed edge called EdgeAccessor::To()/From() → branch_ctx_->ResolveVertex(gid, NEW) (a diff-engine skiplist probe + historical resolve) to get the next vertex — while main just follows the raw Vertex* off the edge. Extending the branched-bit fast-path to To()/From() (un-branched main endpoint ⇒ return it directly, skip ResolveVertex) is what collapsed traversal: shortest-path 3.88×→1.73× (branch 1.04ms→0.42ms), deep 4-hop 3.01×→2.39×. This was the decisive fix, and it confirmed the bottleneck was endpoint resolution, not edge-list retrieval.

Where the remaining overhead lives

  • Full-graph aggregations (~2.3×) — MATCH (n) uses the gid-ordered full-scan union (UnionVerticesIterable), which the scan fast-path doesn't cover. The cost is the O(M) two-stream merge + per-row property read over every vertex; the branched-bit can only skip O(1) per-row checks there, so a fast-path is expected to be marginal (same shape as the dedup hash-set that was a wash). Being probed before committing a build.
  • Deep expansion residual (2.4×) — after the endpoint fix, what's left is the eager vector<EdgeAccessor> materialization per hop + branch-aware accessor overhead + View::OLD reads. Diminishing returns; deeper to attack (lazy edge streaming).
  • Shortest-path residual (1.7×) — same class as above; already a strong result for a 15-hop BFS.

Correctness & safety

  • All changes kept versioning_interpreter 82/82 and versioning_historical_access 10/10 green; branch/main isolation sanity passed on every benchmark run.
  • Branched-bit safety rests on a documented, load-bearing invariant: fast-discard is disabled while a fork pin is held, so the under-lock bit write never races the one unlocked delta_ writer; the fast-path read is a relaxed-atomic load. Re-seed runs under the object lock (checked-out set is per-branch-name → concurrent branches can touch overlapping objects).
  • All work is in the working tree, not committed, not pushed (standing constraint).

Follow-ups

  • Aggregate/union fast-path (deferred — needs careful design). A probe proved the ~2.3× is removable (plain historical-only full scan runs at 1.01×), so it's the union+wrapper overhead, not the inherent scan. BUT the naive fix — yielding un-branched rows as plain (non-branch) accessors — is unsafe: query::VertexAccessor gates both its read fast-path and CowIfNeeded() on the same branch_ctx_, so a plain accessor that is later mutated (MATCH (n) SET …) would write straight into main = corruption. A safe recovery needs decoupling the read-fast-path signal from the mutation-COW gate (a cached read-only flag recovers only the O(1) per-call guard, not the union-merge/construction bulk) or a mutation-safe main-scan + branch-patch restructure. Real project, not a fast-path — deferred.
  • Branched-bit clear-tracking (v2) — never-clear degrades the filter under long-lived heavy branch churn (self-heals on restart). Safe clearing needs per-object refcount or a VersionStore touched-set; danger surface is over-clear → stale-false. Deferred.
  • C3 crash (separate, pre-existing, HIGH) — CaptureBranchCommit iterates a transaction's deltas in reversed slab order; a >292-delta branch transaction can capture SET-before-CREATE → re-checkout/restart MG_ASSERT. Fix: iterate chronologically. Independent of perf.
  • Benchmark caveats: single-instance, single-branch, IN_MEMORY_TRANSACTIONAL, read-only overhead; ratios portable, absolute ms hardware-specific.
