⚡ Graph Versioning v1 — Branch vs Main Performance

pokec_small (10k vertices / 121k edges) · binary HEAD 9659d6e0b RelWithDebInfo · 10 runs · paired branch/main ratio · 2026-07-14

Headline: point/index reads on a churned branch are near-free (~1.1×); pure expansion overhead is modest (~1.15–1.25×); but full index range-scans (~5.5×), whole-graph aggregations (~2.4×), and deep multi-hop expansions on split-adjacency vertices (~3.8×) carry a real branch-read tax. The overhead is a function of how many rows the merge touches, not a fixed per-query cost.

Method

  • One instance, two pinned connections. Session M stays on main; Session B checks out branch bench once and holds it. Separate single-connection drivers (branch checkout is connection-scoped — a shared pool would leak B's checkout into M).
  • Worst-case churn on the branch (applied live, session stays checked out): COW 20% of vertices (property update), tombstone ~2% (DETACH DELETE), add branch-native :Friend edges to a 300-vertex hot set (split adjacency), insert 500 branch-native vertices.
  • Paired ratio, unstable-machine-robust. Within each of 10 runs, M and B are measured interleaved per query parameter; the reported number is the per-run ratio branch/main, then the distribution across 10 runs. Machine-wide transients cancel in the ratio — hence the tight ranges below.
  • Clean vs split decomposition for expansions: clean start vertices are untouched (identical edge set to main → isolates pure branch read-path overhead); split start vertices are churned (COW'd + branch-added edges → true worst case, also more rows).

Where the overhead comes from

  • Point reads (~1.1×) — negligible. An index point-lookup resolves one gid; the branch pays a tombstone-set check + a diff-engine probe. Whether the vertex is COW'd or untouched barely matters.
  • Index range scan (~5.5×) — the worst offender. gender='man' matches ~5,000 vertices. The branch runs the diff-engine index scan and the historical index scan, concatenates, skips tombstoned/COW'd gids, and stable_sorts the union. The cost scales with rows returned, so a wide scan pays the full merge+sort tax. This is the single most important number for read-heavy branch workloads.
  • Full-graph aggregations (~2.4×) — consistent tax. MATCH (n) drives the streaming Vertices() union (historical ∪ diff, tombstone-skipping) over the whole graph; every vertex pays the merge step, and the property variant adds the COW-aware property read.
  • Expansions — pure overhead is small (~1.15–1.25×), split adjacency adds ~15–20pp (~1.37–1.44×). On a branch, ResolveEdges always unions historical + diff incident edges and de-dupes by gid — even for an untouched vertex (that's the clean-case overhead). A split-adjacency vertex has branch-added edges, so the union is non-trivial AND there are more rows to traverse.
  • Deep expansion compounds (4-hop split ~3.8×). The per-hop ResolveEdges union multiplies across hops, and on hot split vertices the fan-out is larger on the branch than on main. Absolute cost is also large (main 27ms → branch 105ms) — deep multi-hop on high-degree vertices is the pattern to watch.

Distribution / confidence

The paired-ratio methodology worked as intended on this unstable box: run-to-run spread is tight for every query (e.g. aggregations 2.28–2.41×, index-scan 5.36–5.68×, expansions within ±0.1×). The ratios are stable signal, not noise. Absolute millisecond values will vary with hardware; the ratios are the portable result.

Caveats & scope

A server-crash bug was found while building this benchmark (unrelated to steady-state read perf, but blocking one measurement path). Replaying a branch's changelog on re-checkout / restart aborts (SIGABRT, branch_engine.cpp:208) when the branch has done SET/label changes on a pre-existing fork vertex — the replay assumes a create-before-mutate that COW'd fork vertices don't have. The benchmark therefore applies churn in a session that stays checked out (live diff engine, no replay), which also measures steady-state read overhead more directly. Filed separately; recommend re-checkout + restart durability tests be added.
What this does and doesn't cover: single-instance, single-branch, single-connection-per-role, IN_MEMORY_TRANSACTIONAL, pokec_small. It measures read overhead on a churned branch. It does not cover: branch write throughput, MERGE cost, concurrent multi-session load, larger datasets (overhead trends should hold or grow with graph size for the scan/aggregate cases), or replication.

Recommendations

  • Branch reads are cheap for point/short-traversal workloads (~1.1–1.4×) — the overlay model is well-suited to "fork, poke around, merge" use cases.
  • Wide index scans and full-graph aggregations on a branch are 2.4–5.5× slower. If branch analytics over large result sets matter, the union-then-stable_sort merge-at-read is the place to optimize (e.g. a true streaming k-way merge instead of collect-then-sort, or short-circuiting when the diff side is empty for a given index).
  • Deep multi-hop traversals on high-degree vertices compound — surface this in guidance for agent/what-if workloads that lean on long expansions.
  • Fix the re-checkout/restart replay crash before the durability story can be trusted for branches that edit existing data.
