Core insight that makes this safe

The historical side of a vertex's incident edges is IMMUTABLE for the branch's lifetime. historical_base_ is main pinned at fork_ts; a branch never mutates main, and its own writes go to the diff engine. So the fork-state adjacency of any vertex never changes while the branch is checked out. That means a cache of the historical incident edges needs no invalidation — the classic cache-staleness trap (the user's own warning) simply does not apply to the historical half.

The diff side is small (a branch touches few of a vertex's edges) and dynamic — so we keep collecting it live every call (cheap: it's the small side, and a diff-engine FindVertex on a tiny engine is fast). No caching of the diff side → no diff invalidation problem either.

What is cached (transaction-independent)

NOT storage::EdgeAccessor objects — those embed a Transaction* that is per-query, so caching them across queries would dangle. Instead cache the raw, transaction-independent pieces from main's frozen adjacency, per branched vertex, per direction:

struct HistoricalIncident {                 // one per fork-state incident edge
  storage::EdgeTypeId type;
  storage::Vertex    *other;                // the far endpoint (main Vertex*, stable under fork pin)
  storage::EdgeRef    ref;                  // the edge (gid for ref-only; Edge* if props-on-edges)
};
// on BranchContext, populated at CowVertex time (eager) for the COW'd gid:
std::unordered_map<storage::Gid,
   std::pair<std::vector<HistoricalIncident>,      // OUT
             std::vector<HistoricalIncident>>>      // IN
   hist_incident_cache_;
// plus the vertex's own historical Vertex* so we don't FindVertex it either.

These pointers/refs are stable because the fork pin keeps main's fork-state objects alive and GC cannot reclaim/relocate them while the branch holds the pin (same invariant the branched-bit write already relies on).

Claims to falsify (for skeptic/logic-verifier)

  • C1 (immutability): a checked-out branch never mutates main's fork-state adjacency, and main's fork-state Vertex*/EdgeRef stay alive+unrelocated for the branch's life (fork pin + GC gate). If main CAN advance under a live branch and GC could reclaim a superseded fork-state edge, the cached other/ref could dangle.
  • C2 (transaction-safety): rebuilding a storage::EdgeAccessor from cached raw parts + the current historical accessor's txn yields an accessor identical in behavior to one produced by historical_base_->FindVertex(...).OutEdges(...).
  • C3 (completeness): every path that reaches ResolveEdges for a vertex WITH a cache entry is still correct; every path for a vertex WITHOUT one falls through to the (correct) Step-A union.
  • C4 (diff-wins + tombstone): preserved — diff collected live first, historical-cache entries skipped if shadowed or tombstoned, exactly as Step A.
  • C5 (props-on-edges=true): a COW'd edge modifies props in the diff engine but keeps its gid; the historical cache still lists it, the live diff side also lists it (diff-wins) → the diff copy shadows the cached historical one. Verify no double-count and that the diff copy wins.

Expected payoff / risk

Removes both FindVertex probes' historical half + the historical adjacency walk + rebuild, per branched-vertex read. Ceiling bounded by the FindVertex/rebuild share (~2–3% self measured). Realistic gain: shortest-path worst 2.04→~1.8–1.9×, deep-expansion worst similar; realistic churn small (few branched vertices). Risk concentrated in C1 (dangling if the immutability assumption is wrong) — hence verify before coding.
