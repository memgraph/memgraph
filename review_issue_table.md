## Allocator Call Map

`DBMEM-11` audit summary:

- `DbAllocateBytes` / `DbDeallocateBytes`: low-level primitives in `memory/db_arena.cpp`. Direct users are `DbAllocate<T>`/`DbDeallocate<T>`, `ArenaMemoryResource`, `PageAlignedAllocator`, `ArenaAwareDeleter`, and `MakeArenaAwareUnique`. These functions deliberately do not infer DB ownership from current TLS during deallocation.
- `DbAwareAllocator<T>`: stateless allocator for transient DB-scoped containers and buffers. Current users include `small_vector` fields on vertices/schema info, GC/deleted-id lists, vector-edge removal lists, stream maps, compressed property buffers, and compressor buffers. Allocation requires a pinned DB arena in debug builds; deallocation is allowed from a different thread because jemalloc records the owning arena per extent.
- `ArenaAwareAllocator<T>`: stateful allocator for long-lived DB-owned structures that outlive a single TLS scope. Current users include main vertex/edge/edge-metadata skiplists, in-memory indices/constraints, trigger stores, and disk edge-import cache skiplists. Debug deallocation checks compare the pointer arena with the allocator's captured arena.
- `ArenaAwareDeleter` / `ArenaAwareUniquePtr` / `MakeArenaAwareUnique`: single-object ownership helpers for objects allocated with an explicit DB arena, currently used by `InMemoryStorage::wal_file_`. The deleter carries the expected arena and validates it in debug builds before destroying/freeing the object.
